## 一、 背景與問題

1. 老闆日常撰寫繁體中文文件時，需頻繁輸入全形標點與特殊符號（如引號「」『』、書名號《》、括號【】、項目符號●、色塊█），此類符號多數不在標準鍵盤上，須切換輸入法之特殊符號表或使用內碼輸入，操作步驟繁瑣。
2. 中文與英文輸入法之間的切換本身即有成本：每次切換輸入法上下文（例如從英文輸入切至中文特殊符號表）打斷輸入節奏，尤其在高頻率撰寫文件的情境下，累積成本可觀。
3. 問題定義：**缺一個常駐於桌面、可全域快捷鍵呼叫的符號面板**，讓老闆在任何前景應用程式中，以滑鼠點擊直接送出常用標點符號，不需切換輸入法或查找內碼表。

## 二、 產品定位

1. 單一使用者（老闆本人）之 Windows 桌面常駐工具，非發布型商業軟體。
2. 定位為「送字工具」而非「文字編輯器」：不儲存、不編輯文字內容，僅將使用者點擊之符號即時送往當前前景應用程式的焦點控制項。
3. 免安裝、免第三方依賴：以 Windows 內建 .NET Framework 4.x runtime 與內建編譯器 `csc.exe` 建置為單一 exe。

## 四、 使用者故事（單一 persona：老闆）

1. 作為撰寫繁體中文文件的使用者，我想按 Ctrl + Alt + / 隨時叫出符號面板，以便在任何應用程式中插入常用標點，而不用中斷打字節奏去查特殊符號表。
2. 作為使用者，我想面板顯示我最後點擊的符號作為回饋，以便確認送出的符號正確（FR-008）。
3. 作為使用者，我想點擊符號後面板不搶走目前應用程式的輸入焦點，以便游標仍留在原本正在編輯的位置（FR-006）。
4. 作為使用者，我想按 Esc 快速把面板收起來，以便不佔用畫面空間，且工具仍在背景待命、下次仍可用 Ctrl + Alt + / 叫出（FR-002、DD-3）。
5. 作為使用者，我想這個工具開機後長期常駐於系統匣，不誤觸重複啟動，以便隨時可用而不必每次手動執行（FR-009、FR-011）。
6. 作為使用者，我想在符號送出異常時有除錯線索可查，以便回報問題或自行排查，而不影響日常使用（FR-013）。

## 五、 範圍

### 5.1 範圍內（v1.0）

1. 單一符號面板視窗：7 個固定符號按鍵（FR-003；v1.2 起括號成組一鍵成對輸入，4 組括號 + 3 單符號），4 欄 2 列排列，UI 比照 Windows 10 小算盤風格（NFR-04）。
2. 全域快捷鍵 Ctrl + Alt + / 顯示／隱藏切換（FR-001，v1.1 依 DD-8 改鍵；v1.2 起主鍵盤 VK_OEM_2 與數字鍵盤 VK_DIVIDE 雙註冊），Esc 於顯示期間隱藏（FR-002）。
3. 點擊符號依焦點控制項類別三路路由送字：EDIT 類 WM_CHAR 直遞、主控台 SendInput、其餘剪貼簿中轉自動貼上，失敗後備 SendInput（FR-004、FR-005、DD-4、DD-9）。
4. 不搶焦點視窗行為（FR-006）、視窗置頂（FR-007）、顯示區回饋（FR-008）。
5. 系統匣常駐（FR-009）、視窗關閉鈕視同隱藏（FR-010）、單一實例防護（FR-011）。
6. 熱鍵註冊失敗警告（FR-012）、環境變數控制之除錯日誌（FR-013）。
7. DPI 感知（NFR-03）、免安裝單一 exe（NFR-01）、`scripts\build.ps1` 零外部依賴建置（NFR-02）。

## 七、 風險與對策

| # | 風險 | 影響 | 對策 |
|---|------|------|------|
| R1 | 全域熱鍵蓋過應用程式內同鍵位快捷鍵 | 目標應用程式之對應快捷鍵失效 | v1.1 已緩解：老闆 2026-07-11 回饋 Ctrl + / 實務使用困難後，依 DD-8 改為 Ctrl + Alt + /，主流應用無此預設綁定，衝突面實質歸零；熱鍵註冊失敗時仍有警告訊息（FR-012），可由系統匣操作 |
| R2 | 視窗顯示期間裸 Esc 被全域攔截：前景應用程式的 Esc（IME 組字取消、關閉選單等）會被吃掉一次並隱藏面板 | 面板隱藏後即恢復，但當次 Esc 動作對前景應用不生效 | 對抗審查 3/3 確認之設計代價，已裁決保留（Boss_Prompt 明定 Esc 關閉、不搶焦點約束下無 KeyPreview 替代路徑）；老闆 2026-07-11 採認「R2 可接受」 |
| R3 | UIPI：無法輸入至以系統管理員權限執行的視窗 | WM_CHAR 與 SendInput 皆被阻擋，靜默失敗 | 除錯日誌可查（FR-013，PUNCTINPUT_DEBUG=1） |
| R4 | SendInput 之 VK_PACKET 對 CJK 字元在注音 IME 開啟時被攔入組字區（老闆 2026-07-11 實機回報「預編譯狀態」，V7 當時未涵蓋此情境） | v1.3 起 SendInput 僅餘主控台路徑與後備路徑，主要目標已改剪貼簿中轉（DD-9），組字問題解除 | 後續異常時以 PUNCTINPUT_DEBUG=1 查路由 |
| R5 | 非標準輸入管線應用（遊戲 raw input 等）可能忽略注入方式 | 該類應用無法接收送出之符號 | 屬 Windows 輸入模型之既有限制，非本工具可解決範圍 |
| R6 | 剪貼簿中轉副作用（DD-9／DD-10）：送字時剪貼簿內容被替換，約 0.5 秒後自動還原 | v1.4.3 起僅備份文字類格式（DD-10 白名單：文字、RTF、HTML），非文字格式（圖形、物件、檔案清單等）不還原，使用者原複製之非文字內容須重新複製；目標應用忙碌延遲處理貼上時，存在貼到還原後內容之競態 | 還原正確已實測（V10、V14-T1）；延遲渲染擁有者不凍結已實測（V14-T2 與 Aphy 建置複測）；連續點擊期間不重拍快照以保住原始內容；剪貼簿被鎖定時自動退回 SendInput 後備 |

## 八、 交接考量

本專案文件（PRD 及後續同套文件）撰寫時以「接手者可能為 Opus 或 ChatGPT 5.5」為前提：

1. 送字路由邏輯（DD-4）之判斷依據與對抗審查結論須完整記載於後續規格文件，不依賴撰寫者記憶；程式碼實作見 `src\Program.cs` 之 `SendSymbolToTarget()` 與 `IsClassicEditClass()`。
2. 建置鏈固定依賴 `C:\Windows\Microsoft.NET\Framework64\v4.0.30319\csc.exe`（見 `scripts\build.ps1`），環境路徑須於後續規格文件明列，可照抄執行。
3. R4 與 V7 已於 2026-07-11 由老闆實機驗證通過結案（驗證紀錄見 SPEC 第十一章與 MODIFY_LOG v1.1），不再屬待辦追蹤項目。
4. C# 語言層級受限於 `csc.exe`（4.0.30319，C# 5 相容），後續維護者修改 `src\Program.cs` 時禁止使用字串插值、`?.`、`nameof` 等 C# 6 以上語法（NFR-05）。
